## Language & Communication: Technical Writing for Engineering

### Overview

Today's challenge focuses on **technical communication** — a critical but often neglected skill for engineers. You'll produce a complete technical document that explains your DMA acquisition system to three different audiences.

---

### Part 1: Multi-Audience Documentation (90 min)

**Task:** Write three versions of the same technical explanation:

#### A) Executive Summary (1 page)
**Audience:** Project manager, funding agency, non-technical stakeholder

**Requirements:**
- No jargon, no acronyms without explanation
- Focus on *why* and *what it enables*
- Include: problem, solution, impact, timeline
- Maximum 400 words

**Example opening:**
> "Our ultrasound imaging system captures data 100× faster than existing methods, enabling real-time tissue analysis during surgery. This document describes the data acquisition module that makes this possible."

#### B) Technical Specification (3-4 pages)
**Audience:** Fellow engineer who might implement or modify the system

**Requirements:**
- Precise specifications: timing diagrams, data rates, buffer sizes
- Architecture diagram (hand-drawn or ASCII art acceptable)
- API documentation with examples
- Error handling and edge cases
- References to datasheets and prior art

**Structure:**
1. System Overview
2. Hardware Interface
3. Protocol Specification
4. Performance Characteristics
5. Integration Guide

#### C) Tutorial Guide (2-3 pages)
**Audience:** New graduate student joining the project

**Requirements:**
- Step-by-step instructions with actual commands
- Common mistakes and how to avoid them
- "Why we do it this way" explanations
- Troubleshooting flowchart
- Links to working code examples

**Tone:** Conversational, encouraging, comprehensive

### Part 2: Visual Communication (60 min)

**Task:** Create three diagrams:

1. **System Block Diagram**
   - High-level: show components, data flow, interfaces
   - Use standard symbols (IEEE or ISO style)
   - Include power/ground, clock domains, data buses

2. **Timing Diagram**
   - Show: trigger → acquisition → transfer → processing → display
   - Include actual timing values (ns, μs, ms)
   - Show parallel operations where applicable

3. **State Machine Diagram**
   - DMA acquisition states: IDLE → ARMED → ACQUIRING → TRANSFER → IDLE
   - Show transitions and conditions
   - Include error states

**Tools:** 
- Hand-drawn → photograph
- ASCII art (for markdown)
- Draw.io (free, browser-based)
- TikZ/LaTeX (if you're ambitious)

1. **Audit existing comments**
   - Find 5 examples of poor comments (vague, outdated, obvious)
   - Rewrite them following best practices

2. **Add module-level docstrings**
   - Every `.py` file should have:
     - One-line description
     - Longer explanation if needed
     - Usage example
     - Author and date

3. **Document a complex function**
   - Choose your most complex function (maybe `dma_acq_isr_handler`)
   - Write Google-style docstring with:
     - Description
     - Args with types
     - Returns with types
     - Raises (exceptions)
     - Example usage

**Example:**
```python

                             calibration: dict) -> dict:
```
