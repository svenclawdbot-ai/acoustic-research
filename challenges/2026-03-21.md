## 🎯 Today's Challenge: Build the Array Control Firmware Module

### Background
You've designed an optimal array configuration with the D-optimal Fisher Information approach. Now it's time to bridge theory and hardware: implementing the firmware that will actually control the ultrasound elements on your ESP32-S3-based pulser/receiver.

The challenge focuses on **real-time array control** — sequencing element firing, managing receive beamforming delays, and streaming data efficiently.

### Tasks

1. **Implement element firing sequencer**
   - Support arbitrary element activation patterns
   - Configure pulse timing (PRI, pulse width)
   - Handle 8+ channel expansion via shift registers
   - Generate proper SPI commands for HV pulser chips (MD1210/TC6320)

2. **Build beamforming delay calculator**
   - Calculate time-of-flight delays for each element
   - Support dynamic focusing at programmable depths
   - Implement delay quantization (match to ADC sample rate)
   - Generate delay lookup table for real-time use

3. **Create DMA-accelerated data pipeline**
   - Configure ESP32-S3 ADC for continuous sampling
   - Set up circular DMA buffer for streaming
   - Implement ping-pong buffering (acquire → process)
   - Target: 20 MSa/s sustained throughput

4. **Add command interface (UART/USB)**
   - JSON-based command protocol
   - Commands: `fire`, `set_gain`, `set_focus`, `get_data`
   - Binary data streaming for efficiency
   - Error handling and status reporting

### Deliverable
ESP32 firmware module (`array_control_firmware.c` + `array_control.h`) that:
- Compiles with ESP-IDF
- Implements element sequencer with configurable patterns
- Calculates and applies beamforming delays
- Streams data via USB/UART at full rate
- Includes test commands for validation

### Success Criteria
- [ ] Firing sequencer correctly controls 8+ elements
- [ ] Beamforming delays calculated within ±1 sample period
- [ ] DMA pipeline sustains 20 MSa/s without overflow
- [ ] Command interface responds in <10ms
- [ ] Tested with simulated hardware (logic analyzer/scope)

### Extension (if time permits)

- Consider using FreeRTOS tasks for command parsing vs. data acquisition
